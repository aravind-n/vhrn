# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`vhrn` runs coding agents ("harnesses") inside a container jailed to the current project directory, with default-deny network egress, so an agent runs without exposing the rest of the host or letting a prompt injection exfiltrate to arbitrary hosts. Claude Code is the harness that ships today; the CLI is harness-agnostic. It's a small Go CLI (the wrapper) plus a Go egress proxy and a few Docker/Make/Bash files, all in one Go module.

- `cmd/vhrn/main.go` + `internal/vhrn/` — the CLI (Go, package `vhrn`). It is subcommand-first: `vhrn install <harness>` pulls images and wires a shell alias, `vhrn <harness> …` runs an agent in the box, `vhrn uninstall`/`list`/`net`/`help` manage the environment. It orchestrates but shells out to rsync/cp/gh and the engine to match the original shell behavior.
- `image/base/` (`Dockerfile` + `entrypoint.sh`) — the shared `vhrn-base` image (`debian:trixie-slim` + python3/uv, mise, gh, ripgrep/fd, zip/unzip, nftables; non-root `dev` user, no sudo; the egress entrypoint). No agent.
- `image/claude/Dockerfile` — the `vhrn-claude` image: `FROM vhrn-base` plus the native Claude Code binary. Adding a harness is a spec in `internal/vhrn/harness.go` plus a thin `FROM vhrn-base` Dockerfile here.
- `proxy/` — a hand-rolled Go CONNECT/HTTP egress proxy (static binary in a `scratch` image) enforcing the domain allowlist. Its own dependency-free module (standard library only), published as `vhrn-proxy`. Policy files live host-side, mounted only into the proxy.
- `Makefile` builds the images (`make`), builds/installs the Go binary (`make binary` / `make install`), cross-compiles releases (`make release`), and runs tests (`make test`). Build artifacts (the binary and cross-compiled releases) go under `out/`. `scripts/install.sh` is the curl installer that fetches a prebuilt binary from GitHub Releases.

## Commands

- `make` / `make build` — build all images (`build-base` then `build-proxy` then `build-claude`; the harness image is `FROM vhrn-base`, so base builds first). `make build-base`/`build-claude`/`build-proxy` — one image. `make clean` — remove the images and the built binary.
- `make binary` — build the static `vhrn` binary to `out/vhrn`; `make release` — cross-compile darwin/linux × arm64/amd64 into `out/dist/`.
- `make test` — all unit tests: `go test ./...` (CLI) plus `cd proxy && go test ./...` (the proxy's own module). Tests cover flag parsing, the history-key encoding, terminal env, allowlist add/dedup, engine-inspect IP parsing, the harness registry, the installed registry, shell-alias blocks, install/uninstall arg assembly, the persistence state store (creds bootstrap + `.claude.json` merge), the mount topology, TOML config load/merge, `blocked_dirs`, net-mode resolution, and toolchain hashing — plus the proxy's allowlist-matching and IP-classifier tests.
- `make ENGINE=docker ...` — force Docker; the engine auto-detects `container` first, then `docker`.
- `make install` — build and install `vhrn` to `/usr/local/bin` (needs sudo); then `vhrn install claude` pulls the images and adds the alias (`--local` uses `make build` images). `make uninstall` removes the binary.
- No CI yet, and the unit tests don't exercise a live container. To verify the full run path, `vhrn install claude` (or `make`) then run `vhrn claude` in a throwaway project directory.

## Must-know invariants

- **Both Apple `container` and Docker must work, for build *and* run.** The Makefile, install (`internal/vhrn/install.go`), and run (`internal/vhrn/engine.go`) all select the engine (explicit `ENGINE`/`VHRN_ENGINE`, else auto-detect `container` then `docker`) — keep them in sync. The CLIs differ (`container image delete` vs `docker image rm`; inspect output differs, and Apple escapes the CIDR slash in `ipv4Address`), so an engine switch isn't a pure string swap. The box must run with `--cap-add CAP_NET_ADMIN` or the entrypoint's `nft` fails with a netlink permission error.
- **The wrapper is a thin pass-through for the agent.** `vhrn <harness> [wrapper-flags] [--] [agent args]` consumes only its own flags (`--open-net`/`--allow`), then forwards the rest to the agent verbatim. Don't bake agent flags in. Bare `vhrn` prints help — a deliberate clean break from the old `vhrn <claude args>`.
- **Harnesses are data, not forks.** `internal/vhrn/harness.go` holds the registry; a `Harness` spec carries the image name, in-box command, alias, default egress domains, and the persistence descriptors. Dispatch, install, run, and persistence all read from it. Adding codex/aider = a spec, a `FROM vhrn-base` Dockerfile under `image/<harness>/`, and a CI matrix entry.
- **Login/state persists via a box-owned store, not the disposable copy.** `~/.cache/vhrn/state/<harness>/` is mounted as the box's base config dir, and the agent is pointed at it (`CLAUDE_CONFIG_DIR` for claude). Host credentials are copied in **only when the store is empty** (bootstrap-only — an in-box login is never overwritten); `.claude.json` is merged in place to set onboarding + this project's trust without touching `oauthAccount` or other projects. The disposable synced config (skills/commands/agents/settings/statusline), the box guide, and the `projects/<key>` history layer on top as **nested** mounts, so the config sync can never reach `state/`. Claude rewrites `.claude.json` via a backup file, so it must sit in a real directory mount (verified: setting `CLAUDE_CONFIG_DIR` relocates `.claude.json` into that dir) — never a single-file mount.
- **The container stays ephemeral (`--rm`).** That is a security feature — a fresh, tamper-proof firewall every boot. Persistence is a property of what's mounted; do **not** move to a persistent "container machine."
- **Terminal env crosses verbatim.** `TERM`/`COLORTERM`/`TERM_PROGRAM`/`TERM_PROGRAM_VERSION` are forwarded as-is, never forced. Don't reintroduce `COLORTERM=truecolor`/`FORCE_COLOR`.
- **The disposable config copy (`~/.cache/vhrn/sandbox`) is re-synced from `~/.claude` on every run** (`rsync -aL --delete`, `cp -RL` fallback), so edits made there are wiped each run — change `~/.claude` instead. It is physically separate from `state/`.
- **The history key must match Claude's own `projects/<key>` encoding** (`[^A-Za-z0-9]` → `-` on the absolute project path). Keep it in lockstep or in-box history stops unifying with native history.
- **gh auth is env-injected, never file-mounted.** The wrapper resolves a token (`$GH_TOKEN`/`$GITHUB_TOKEN`, else `gh auth token`) and passes it as `GH_TOKEN`; the entrypoint runs `gh auth setup-git`. Skips silently when the host has no gh login. SSH remotes stay unauthenticated by design.
- **The host `~/.gitconfig` is copied into the cache and bind-mounted at `/home/dev/.gitconfig`** (skipped if absent). A disposable copy — change the host file.
- **The egress guard is enforced from outside `dev`'s reach.** The entrypoint installs a default-deny nftables ruleset as root (egress only to the proxy), then drops to `dev` via `setpriv`; because the uid transition clears capabilities and there is no sudo, `dev` cannot alter the firewall. This is *why* sudo was removed — don't reintroduce it. If `nft` can't run, the entrypoint aborts rather than fall through to an unguarded session.
- **Egress policy is host-owned.** The allowlist, mode, and deny-log files live in `~/.cache/vhrn/net/` and are mounted only into the proxy, never the box. `vhrn net …` mutates them from the host; `install` unions base + harness default domains into the allowlist (append-if-missing); config `net.allow`/`net.mode` fold in at run (`--open-net` wins). `net allow` writes atomically (same-dir temp + rename). The box can never widen its own egress.
- **Images are pulled from a registry, not built by users.** `vhrn install <harness>[@version]` pulls `vhrn-<harness>` and the same-version `vhrn-proxy` from `ghcr.io/aravind-n` (override `VHRN_REGISTRY`); CI (`.github/workflows/publish-images.yml`) builds and pushes them on a git tag (multi-arch). Image and binary versions are **decoupled** — a user pins or rolls back a version without updating the binary. `--local` uses `make build` images (recorded as version `local`). The installed registry (`~/.config/vhrn/installed`, `name version` per line) is the source of truth the run path resolves the image ref from; an uninstalled harness aborts the run. The box and its proxy always run the same version (a matched set).
- **Config precedence is flags > project `./.vhrn.toml` > global `~/.config/vhrn/config.toml` > defaults** (`internal/vhrn/config.go`, parsed with `github.com/BurntSushi/toml` — the CLI module's only third-party dep; the proxy stays stdlib-only). `blocked_dirs` matches the resolved cwd **exactly** (not subtree) so projects under `$HOME` still run; default `["~","/"]`. `toolchains.tools` resolves to a content-addressed derived image (`vhrn-<h>-tc-<hash>`, `FROM` the harness image + `mise use -g`), built once and cached by tag.
- **Shell aliases and the installed registry are host state.** `~/.config/vhrn/installed` is the source of truth; `install`/`uninstall` mutate it and regenerate reversible marker-delimited alias blocks in bash/zsh/fish rc files (only files that exist plus the current shell's). `command <name>`/`\<name>` still reach the real binary.
- Claude Code is the native binary in `~/.local` (baked into the image; no host install needed) and honors `HTTPS_PROXY`. The entrypoint clears a stale `$PWD/.git/index.lock` on boot (needs `procps`).

## Conventions

- The CLI is Go (`cmd/vhrn/main.go` + `internal/vhrn/`, package `vhrn`); it orchestrates and shells out to rsync/cp/gh and the container engine. Comments explain *why*, terse; prefer small single-file helpers over new packages. Keep pure logic (arg assembly, hashing, merges, matching) in testable functions.
- Bash/sh scripts (entrypoint, scripts/install.sh) use `#!/usr/bin/env bash`/`sh` and `set -euo pipefail`; comments explain *why*, terse — one line where possible. Helper functions early-`return 0` when a source path is absent.
- Commit messages: Linux-kernel style (`cli: …`, `image: …`, `Documentation: …`), concise and imperative.
